# ADR 005: snake_case field names throughout TypeScript types

**Status:** Accepted  
**Date:** 2026-06-15

## Context

When the backend was Django REST Framework, DRF serializers returned JSON with snake_case field names (e.g., `gross_income`, `growth_rate`). The old `useApi` composable applied `toCamelCase` to convert responses to camelCase, so all TypeScript interfaces used camelCase (e.g., `grossIncome`, `growthRate`). A `toSnakeCase` conversion was applied on write.

After replacing Django with Supabase, data flows through PostgREST, which returns column names exactly as defined in the schema — snake_case. The conversion layer is now dead weight and a source of bugs (a field missed in conversion would fail silently).

## Decision

All TypeScript interfaces and type definitions in `types/` use snake_case field names matching the Supabase schema column names exactly. The `toCamelCase` / `toSnakeCase` utility functions are removed.

```typescript
// Before
interface Income {
  grossIncome: number;
  growthRate: number;
  incomeType: IncomeType;
}

// After
interface Income {
  gross_income: number;
  growth_rate: number;
  income_type: IncomeType;
}
```

`types/database.types.ts` is auto-generated by `supabase gen types typescript --local` and is the source of truth. Hand-written types in `types/` must match it.

## Consequences

**Positive:**
- No conversion layer — PostgREST data flows directly into typed objects
- Types are validated against the DB schema (the generated `database.types.ts` reflects the actual columns)
- Eliminates an entire class of "field not found" bugs caused by missed conversions

**Negative / Trade-offs:**
- Breaking change — every consumer of domain types must be updated (components, composables, stores, pages)
- PascalCase component props and template bindings still use camelCase per Vue convention; only the DATA model uses snake_case, not template attributes or computed property names
- The simulation engine (`models/`) uses camelCase internally for computation variables. Config interfaces it accepts must now be snake_case, but internal variables like `let growthRate = config.growth_rate` remain camelCase internally — this is intentional (simulation logic reads snake_case from data, maps to local variables for computation).

## Guidance for future types

- Always match the column name exactly as it appears in the SQL migration
- Do NOT invent camelCase aliases — they will drift from the schema
- Run `supabase gen types typescript --local > types/database.types.ts` after any migration and check the diff
- Template/component props may still use camelCase (Vue convention), but when they represent DB fields, the underlying data property is snake_case
